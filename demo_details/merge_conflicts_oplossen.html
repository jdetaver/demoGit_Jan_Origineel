Mergeconflicts oplossen

Wat is een merge conflict ?

Merge conflicten treden voornamelijk op zodra verschillende mensen aan een project samen werken. Een merge conflict treedt op als git er niet in slaagt om 2 repository's samen te voegen. Git vraagt dan assistentie om bij het behouden van code.

Voorbeeld :

  1. Twee personen werken aan een file. Bij de start van de werkdag halen ze de remote repository nog even binnen (pull). Beide hebben dus dezelfde startpositie.
  2. Persoon A past de file aan van "Dag VDAB" naar "Dag PHP"
  3. Persoon B past zijn file ook aan : van "Dag VDAB" naar "Dag JS"
  4. Beide personen doen een commit zodat hun eigen lokale repository aangepast is.
  5. Op het einde van de werkdag stuurt persoon A zijn repository naar de remote repository (push)
  6. Deze ondervind daarbij geen problemen. De remote repository weet namelijk wat moet gebeuren : wijzigen "VDAB" in "PHP"
  7. Nu zal persoon B zijn wijzigen doorsturen naar de remote repository. De wijziging zou moeten zijn : wijzig "VDAB" in "JS". Deze actie is echter onmogelijk aangezien de remote repository reeds verder geÃ«volueerd is. Er ontstaat dus een merge conflict.

Hoe los ik ze op ?

Zodra git een merge conflict detecteert zal VSCode beide aanpassingen tonen. Je keuze kan je gewoon aanklikken en VSCode doet de aanpassing in jouw lokale repository. De keuzen zijn voor de hand liggend :

  • Accept Current Change : behoudt jouw (groen) aanpassing en laat de andere aanpassing vallen
  • Accept Incomming Change : behoudt de aanpassing van de remote repository (blauw) en laat jouw aanpassing vallen
  • Accept Both Changes : behoudt beide aanpassingen.
  • Compare Changes : toont beide aanpassingen in de file op een andere manier

Het is mogelijk dat je eerst nog de aanpassingen van de staging area naar de lokale repository moet brengen Indien dat het geval is moet je nog een commit doen.

Nu kan je jouw lokale repository doorsturen naar de remote zonder dat dit nog problemen zal opleveren. Klik hiervoor op de "Sync Changes"-knop.
